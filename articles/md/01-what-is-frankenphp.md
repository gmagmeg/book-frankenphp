# FrankenPHPとは何か

## 1.1 はじめに
FrankenPHPは、Go言語製Webサーバー「Caddy」にPHPランタイムを統合したPHPアプリケーションサーバーで、作者はKévin Dunglas、初回リリース日は2023年です。公式ソースは `https://github.com/php/frankenphp` で、GitHubのURLをよく見るとわかるのですが、PHPの公式リポジトリとして扱われています。これは2025年5月、PHPを支援する非営利組織 The PHP Foundation からの公式サポートが公表され、今日ではPHP実行環境の有力な選択肢の1つとして、注目されています。
出典: https://thephp.foundation/blog/2025/05/15/frankenphp/

## 1.2 PHP実行環境の変遷とFrankenPHPのアプローチ
FrankenPHPが登場するまでのPHP実行環境の流れを、大きく2つの代表的な構成に絞って振り返ります。ここを押さえると、FrankenPHPが「何を解決しようとしているのか」が見えやすくなります。

### `Apache + mod_php`
初期に広く使われたのが `Apache + mod_php` です。この構成では、WebサーバーであるApacheのプロセス内でPHPを直接実行します。構成が分かりやすく導入しやすい一方で、アクセス増加時のリソース効率や運用の柔軟性に課題が出るケースがありました。特に同時接続数が増える状況では、いわゆるC10k問題（1万接続規模での性能・資源効率の課題）が注目され、接続ごとに重い処理資源を消費しやすい構成は不利になりがちでした。
この背景から、より少ない資源で多くの同時接続を捌きやすいアーキテクチャが求められるようになります。

### `Nginx + PHP-FPM`
`Apache + mod_php`の課題に対応する形で普及したのが `Nginx + PHP-FPM` です。Webサーバー`Nginx`とPHPの実行基盤である`php-fpm`を分離し、両者をFastCGIで接続します。この分離によって、同時接続への対応やプロセス管理の柔軟性が高まり、実運用での扱いやすさが向上しました。
しかしアプリケーションの規模拡大と運用の高度化が進んだことで、状況が変わってきます。Webサーバーと実行環境基盤を切り分けていたことで、コンテナ運用やオートスケールが一般化し、構成要素が増えるほど、設定管理や障害切り分けの負担が大きくなってきたのです。その結果、`Nginx + PHP-FPM` は依然として有力な構成である一方、FastCGIを挟む構造に由来する複雑さや運用コストを見直す動きも強まっていきました。

### `FrankenPHP` のアプローチ
FrankenPHPは、`Nginx + PHP-FPM` で得られた運用知見を踏まえつつ、Webサーバーと実行環境を1つにまとめ、`Apache + mod_php` のようなWebサーバー一体型のアプローチを取りました。これによりFastCGIを介したプロセス間通信が生む複雑さやオーバーヘッドを減らす方向で設計されています。また`Apache + mod_php` が課題とした、同時接続時の資源効率や運用の柔軟性を踏まえ、FrankenPHPはCaddyの構成言語であるGoを軸に、現代的に再構成されています。
つまり、`Apache + mod_php` のメリットであった統合型のシンプルさと、近年の運用で重視される実用性を両立させようとするのが、FrankenPHPの立ち位置です。
